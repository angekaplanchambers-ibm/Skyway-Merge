## What Changed

Option B should not imply a multi-org dashboard layer. The corrected comparison is:

- Option A adds an org dashboard before the org.
- Option B starts at a single org and keeps workspace as the main hub.

## Current Baseline

```text
HCP
└─ Org
   └─ Dashboard
      └─ Project
         ├─ Infragraph
         ├─ IAM
         └─ Settings

Terraform Explorer
└─ Org
   └─ Workspaces
      └─ Explorer View
         └─ Search / Import
```

## Option A - Long-Tail Org-First

Use this if the goal is to make org the main container for all infrastructure views.

```text
Org Dashboard
└─ Org
   ├─ Explorer View
   ├─ Workspace
   │  ├─ Search
   │  └─ Import
   ├─ Infragraph
   ├─ IAM
   └─ Settings
```

### Read on Option A

- Adds a new org dashboard layer.
- Makes org the place where users choose between explorer, workspace, graph, IAM, and settings.
- Keeps Search and Import inside Workspace.

## Option B - Skyway Workspace-Centric

Use this if the goal is to avoid a new multi-org dashboard and keep workspace as the main operating surface.

```text
Org
└─ Workspaces
   ├─ Search
   ├─ Import
   ├─ Explorer View
   ├─ Infragraph
   ├─ IAM
   └─ Settings
```

### Read on Option B

- No multi-org dashboard.
- Entry point is the org.
- Workspaces becomes the main hub for the rest of the tools.
- This is closer to the current Terraform mental model.

## Shared Constraints From The Workshop

- Search and Infragraph overlap on discovery and query work.
- Explorer View and Infragraph overlap on managed resource visibility.
- Import stays Search-led.
- Explorer View stays the managed-resource and on-prem surface.
- Infragraph stays the relationship and dependency surface.

## Simple Comparison

```text
Option A
Add a new org dashboard and fan out from org.

Option B
Do not add a new dashboard. Start at org and make workspace the hub.
```
